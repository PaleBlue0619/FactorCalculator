# 引言
随着市场的发展，用于描述市场的因子数量急剧增加，传统的Python计算和DataFrame格式已无法满足因子回测对计算性能的需求。并且，因子的有效性也将随着量化金融领域博弈程度的加剧而减弱。因此，如何设计并使用更高效的流程和工具，以更快速地发现新的有效因子，已成为每个交易团队及个人必须面对的市场挑战。
现阶段，开源的因子计算框架缺乏整合的体系，导致实际代码开发流程零散，后期修改模型与因子定义也较为繁琐，主要问题可以归纳为以下几点：
1. 如果原始数据源的结构发生了列名的修改，往往意味着所有依赖该列的因子表达式均需进行变更；
2. 同时，如果存在多表关联计算因子的情况，现有框架的解决方案是提前拼接好，或是进行SQL联邦查询，但在现有表数量较多的时候，大量的排列组合会占据大量内存，进而诱发OOM；
3. 现有框架的因子表达式解析往往采用逆波兰表达式Stack或公式树的形式，但算子的第三方库强依赖使得其拓展性较差，同时无法实现降频因子的聚合计算，因而对复杂的因子表达往往无能为力。
4. 此外，因子表达式之间的关系往往被业界所忽略，导致新因子往往需要从头计算，而浪费了宝贵的中间结果，使得整体的运算效率降低，同时使得研究员在撰写新因子时通常需要从头写起，当原始因子的表达式发生变动时，所有新因子的相关代码部分也需要进行修改；
5. 同时，随着因子数量的增加，传统DataFrame的存储模式已不能满足批计算性能的需求，且在实时计算方面，若框架未能解析因子表达式中的callBackPeriod，将因为窗口区间难以确定使得因子计算结果出现错误。

因子DAG调度计算框架(FactorCalculator)的功能是由用户根据因子之间的依赖关系书写因子表达式，包括自定义该因子依赖的相关函数、该因子所需的分钟频、日频数据源信息等。FactorCalculator框架会自动根据配置文件构建DAG，并利用拓扑排序、深度优先遍历（DFS）等图论算法进行调度计算，以实现因子计算的正确性、中间结果的最大利用与流批一体的计算效果。因子DAG调度计算框架会自动将计算出的分钟频、日频因子分别存放在两张 DolphinDB dfs 分布式表中。
